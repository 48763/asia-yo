## 03

EC2 無法 ssh 連線訪問，這個行動我會分下面三個階段: 
- [進入階段](#進入階段): 嘗試各種方式進入機器。
- [排查階段](#排查階段): 進入之後，判斷事故根因。
- [復原階段](#復原階段): 根據事故根因，決定復原方式。

### 進入階段

#### 階段 1 

先在本機跑 `ssh -vvv`，看是否卡在下面問題:
- Connection refused: sshd 可能掛了 - ***跳到[階段 2](#階段-2)***
- Permission denied: key 或權限問題 - ***問題如果是這些，通常沒什麼大礙***
  1. Key 權限過大，就把權限設 `600` - ***輕鬆解決***
  2. `~/.ssh/config` 設定錯誤，再確認設定是否都正確 - ***輕鬆解決***
  3. 遠端的 `authorized_keys` 沒有加入 public key - ***跳到[階段 2](#階段-2)自己加入，如果沒安裝，且沒人能支援，只能 DR ...***
- Closed by remote host: 排除網路問題，可能的是 `/home` 權限被改掉，導致 sshd 阻擋 - ***跳到[階段 2](#階段-2)***
- Timeout: 可能磁碟滿了或負載太高 - ***跳到[階段 2](#階段-2)***

#### 階段 2

> 假設沒有安裝 SSM 和沒有開啟 Serial Console，將跳到[階段 3](#階段-3)

因為 SSH 進不去，我會依序嘗試下面方式:
- 如果有安裝，就用 SSM 進去
- 沒有或無效，就用 EC2 Serial Console

> 如果可以進入，就進入[排查階段](#排查階段)

#### 階段 3

最後一招 - 先 snapshot ，再從快照復原硬碟，最後掛到乾淨的機器，然後進入[排查階段](#排查階段)。

### 排查階段

進入機器後，會排查是否是下面問題:
- sshd: 使用 `systemctl status sshd` 等相關指令，看是否有異常退出
  - 有的話，會看系統的資源監控，看是否是因為資源不足導致。
- 磁碟空間: 使用 `df -i`、`df -h`，查看是否是磁碟或索引滿了
  - 是的話，再使用 du 指令找出哪個目錄佔用空間過大，再安排清理計劃。
- 系統設定問題:
  - 權限: 確認 `/home`、`~/.ssh`、`~/.ssh/authorized_keys` 權限是否正確。
  - ssh: 確認 `/etc/ssh/sshd_config` 設定是否正確。
- 機器效能問題: 調整資源配置(短解)與優化應用程式(長解)

### 復原階段 

除了可以透過 [階段 1](#階段-1) 或 [階段 2](#階段-2) 進入機器，並解決簡單配置問題，其它可能都需要修復計畫。

但操作都不外乎就是:
- 建立新環境
- 測試新環境
- 切換流量到新環境
  - DNS
  - LB
